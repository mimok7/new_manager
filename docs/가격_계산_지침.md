## 가격 계산 지침

목적: 가격 계산 로직을 중앙화하여 유지보수·정책 변경을 한 번에 반영하도록 합니다.

1) 핵심 원칙
- Single Source of Truth: 모든 가격 계산 로직은 `packages/domain/src/pricing` 내에 위치시킨다.
- 앱은 직접 DB 쿼리나 계산식을 구현하지 않고, `@sht/domain/pricing`를 import 해서 사용한다.
- UI(페이지)는 계산 결과만 소비한다. 계산 책임은 domain 패키지가 가진다.

2) 디렉터리 구조 (권장)
- `packages/domain/src/pricing/`
  - `cruise.ts` (기존 구현 유지)
  - `hotel.ts` (호텔 가격 계산기)
  - `airport.ts` (공항 가격 계산기)
  - `rentcar.ts` (렌트카 계산기)
  - `tour.ts` (투어 계산기)
  - `codes.ts` (가격 코드 조회 헬퍼)
  - `sht.ts` (좌석 단가 계산)
  - `index.ts` (통합 export)

3) 사용 규칙 (예시)
- 프론트엔드 파일에서 직접 DB 조회/계산 금지. 예:

```ts
// OK: 중앙 계산기 사용
import { CruisePriceCalculator } from '@sht/domain/pricing';
const calc = new CruisePriceCalculator(supabase);
const result = await calc.calculate({ /* input */ });

// NG: 앱 내부에 동일한 calculate 구현 금지
```

4) 마이그레이션 전략 (단계별)
- 1단계: `packages/domain/src/pricing/cruise.ts` 의존성 확인 및 통합 export 유지
- 2단계: 앱 내부의 복사본(`apps/*/lib/cruisePriceCalculator.ts`)을 얇은 래퍼로 교체:
  - 기존 API와 동일한 export를 유지하는 파일로 바꿈(예: `export * from '@sht/domain/pricing'`).
- 3단계: 각 앱에서 래퍼 호출을 테스트하고 문제가 없으면 중복 파일 삭제
- 4단계: 호텔/공항/렌트카 등 추가 계산기 동일 방식 적용

5) 캐싱 & 성능
- 클라이언트: React Query 훅에서 `staleTime`/`cacheTime` 설정 권장(예: 10분)
- 서버: 빈번한 계산은 DB 기반 캐시 테이블 또는 Redis 고려

6) 테스트
- 각 계산기 파일에 unit tests 작성 (vitest 사용 권장)
- 경계값(유아 무료 기준, 엑스트라베드 등) 케이스를 포함

7) DB 요구사항
- 현재 테이블로 충분: `cruise_rate_card`, `cruise_holiday_surcharge`, `hotel_price`, `airport_price`, `rentcar_price`, `tour_pricing`
- 선택사항: 가격 변경 이력(`price_change_log`) 테이블 생성 고려

7-1) 예약별 가격 수정 저장/표시 원칙
- 기준 가격표 테이블(`hotel_price`, `airport_price`, `rentcar_price`, `tour_pricing`, `cruise_rate_card`)은 예약 수정 화면에서 변경하지 않는다.
- 예약별 수동 단가와 서비스 총액은 각 예약 상세 테이블(`reservation_hotel`, `reservation_airport`, `reservation_rentcar`, `reservation_tour`, `reservation_cruise_car`)에 저장한다.
- 예약별 추가요금, 할인, 최종 합계, 계산 내역은 `reservation.price_breakdown`에 `reservation_pricing_v1` 구조로 저장한다.
- 화면 표시, 결제 상세, 확인서, 관리자 상세 화면은 기준 가격표 합산보다 `reservation.total_amount`와 `reservation.price_breakdown.grand_total`을 우선 표시한다.
- 추가요금 내역은 `reservation.manual_additional_fee`와 `reservation.manual_additional_fee_detail`을 우선 표시한다.
- `apps/manager` 예약 수정 기능을 바꿀 때 동일 경로가 `apps/manager1`에 있으면 반드시 같은 변경을 반영한다.
